# Bundle Adapters

AI-NEMO converts resource-bundle files to and from the
:class:`~ainemo.core.segment.Segment` data model via **bundle
adapters**. Cycle 1 ships four adapters covering the dominant software
i18n formats. Adding a new format means writing one adapter that
implements the :class:`~ainemo.core.adapters.base.BundleAdapter`
Protocol — no other layer of the pipeline changes.

## Adapter Protocol

Every adapter exposes:

| Surface | Description |
|---|---|
| `format_id: ClassVar[str]` | Stable kebab-cased identifier (e.g. `"java-properties"`). |
| `file_extensions: ClassVar[tuple[str, ...]]` | Extensions including the leading dot (e.g. `(".properties",)`). |
| `parse(path, source_lang) -> tuple[Segment, ...]` | Read the file; return Segments in original key order. |
| `serialize(path, translated, target_lang) -> None` | Write the translated bundle in the same format. |

Round-trip identity holds at the segment-list level: `parse → serialize → parse` produces the same Segment list (modulo translated text) for every adapter-specific fixture. File-level byte identity is **not** guaranteed — adapters may normalize whitespace, key ordering inside groups, or comment placement.

## `JavaPropertiesAdapter` — `.properties`

Use for JVM resource bundles, Spring Boot `messages_*.properties`, and
the rest of the Java-shaped i18n world.

```python
from ainemo.core.adapters.java_properties import JavaPropertiesAdapter
adapter = JavaPropertiesAdapter()
segments = adapter.parse(Path("messages_en_US.properties"), source_lang="en-US")
```

**Supported features**
- `=`, `:`, and whitespace separators
- Line continuations (trailing `\`)
- Standard escapes: `\n`, `\t`, `\r`, `\f`, `\\`, `\=`, `\:`, `\#`, `\!`, `\u####`
- UTF-8 native and `\uXXXX` escapes (parser decodes both; serializer emits UTF-8 directly)
- Comments (`#` and `!`) — preserved on the next-following key's `Segment.metadata["comment"]`

**Cycle-1 limitations**
- Trailing comments after the last key are lost in round-trip (no Segment to attach them to).
- Mixed `=`/`:`/whitespace separators are normalized to `=` on serialize.
- ISO-8859-1 `\uXXXX`-escape output mode is not implemented; output is always UTF-8.

## `I18NextJsonAdapter` — i18next JSON

For the JavaScript i18next ecosystem (React, Vue, Next.js).

**Format flexibility**
- Flat keys: `{"login.button.submit": "Submit"}`
- Nested keys: `{"login": {"button": {"submit": "Submit"}}}`

The parser flattens both shapes to dot-keys internally. The serializer always emits **nested** JSON (the i18next convention). Round-trip identity is therefore segment-list-level: same flat keys → same nested output regardless of input shape.

**i18next plurals**: each suffix variant (`_zero`, `_one`, `_other`) is its own translatable Segment. Cycle 1 does not branch-decompose them — that's cycle 2+ work. The ICU plural form (`{count, plural, …}`) is the canonical plural-handling path; it parses through `core/icu.py` like any other ICU placeholder.

## `GettextPoAdapter` — `.po`

For the GNU gettext family — Python (Babel/Django/Flask-Babel), WordPress, Drupal, most non-Java OSS.

Backed by [polib](https://github.com/izimobil/polib).

**Plural entries — cycle-1 scope: 2-form source, 2-form target.** gettext source PO files always carry exactly two source forms — `msgid` (singular) and `msgid_plural` (plural). The adapter produces two Segments per plural entry, keyed `"<base>#0"` and `"<base>#1"`, that translate independently. Serialization writes the two corresponding target translations to `msgstr[0]` and `msgstr[1]`. **This is correct only for target languages with two plural categories** — English, German, Dutch, French, Spanish, Italian, Portuguese, Hebrew (post-CLDR-update), and most Western European languages.

Target languages with more plural categories — Russian (4: one, few, many, other), Polish (3: one, few, many), Arabic (6), Czech (3) — are **NOT FULLY SUPPORTED in cycle 1**. The adapter emits only `msgstr[0]` and `msgstr[1]`; the additional plural forms required by the target language's `Plural-Forms` header rule are missing from the output PO. Most gettext runtimes fall back to `msgstr[1]` for unfilled forms — technically under-specified output. Cycle 3+ revisits this once the Provider Protocol gains target-language CLDR plural-rule awareness; the cycle-1 serializer already passes through forms 2..N when supplied (pinned by `tests/unit/test_gettext_po_adapter.py::test_serialize_passes_through_n_form_plurals`), so cycle-3+ generation is purely caller-side work.

**Comments preserved across round-trip**: translator (`# `), extracted (`#. `), reference (`#: `), flags (`#, `).

**Other cycle-1 limitations**
- The default PO header generated by `serialize` is minimum-viable (Content-Type + Language); preserving the source file's full header (Project-Id-Version, POT-Creation-Date, etc.) is cycle-2 work.
- Obsolete entries (`#~ msgid`) are skipped on parse — translation-time work doesn't touch them.

## `XliffAdapter` — XLIFF 2.0

Bilingual interchange for translation tooling integrations. XLIFF 1.2 is intentionally out of scope — the formats differ enough that a single adapter would tangle.

**Mapping**
- `<unit>` + `<segment>` → one Segment per `<segment>`, keyed `"<unit_id>#<segment_id>"`.
- `<file id>` and `<group id>` preserved on `Segment.metadata` so the unit hierarchy round-trips.
- `<note category="…">` preserved per-category on `Segment.metadata` (key prefix `xliff.note.`).

**Inline markup — NOT supported in cycle 1.** XLIFF 2.0 allows inline elements (`<mrk>`, `<ph>`, `<sc>`, `<ec>`) inside `<source>` and `<target>`. The cycle-1 parser collects only the text content — inline child tags are dropped on parse with a `DEBUG`-level log entry naming the affected unit. Files that rely on inline codes for placeholder/format preservation lose that markup through the cycle-1 round-trip.

The cycle-1 design rationale: an earlier attempt preserved inline children as XML strings inside `Segment.source_text`, but lxml escapes those strings on serialize (`<ph id="x"/>` → `&lt;ph id="x"/&gt;`), emitting silently-broken XLIFF. Dropping is the cycle-1-honest choice. Cycle 2+ rebuilds inline children as real XML nodes on serialize and exposes them through a new `Placeholder.kind` value to the parser. Pinned by `tests/unit/test_xliff_adapter.py::test_parse_drops_inline_markup_cleanly` and `::test_serialize_does_not_escape_brackets_in_text`.

## Auto-detection

The CLI infers the adapter from the file extension by default; pass `--format` to override:

```bash
nemo translate --from messages.txt --format java-properties --to-langs de-DE
```

| Extension | Inferred adapter |
|---|---|
| `.properties` | `java-properties` |
| `.json` | `i18next-json` |
| `.po` | `gettext-po` |
| `.xlf`, `.xliff` | `xliff-2` |

## Adding a new adapter

1. Create `src/ainemo/core/adapters/<name>.py` with a class that implements the `BundleAdapter` Protocol.
2. Add ≥10 fixtures including pathological cases (Unicode keys, escape sequences, multi-line values, empty values, comments).
3. Ship a `test_<name>_adapter.py` with the same shape as the four existing adapter tests: Protocol conformance, parse happy paths, edge cases, serialize, target-lang mismatch rejection, and a round-trip-identity property test parametrized over the fixtures.
4. Register the adapter's `format_id` and `file_extensions` in `src/ainemo/cli/commands.py:_ADAPTERS`.
